# 면접 평가 시스템 설계

## 1. 설계 목적

AI 면접관 에이전트가 지원자의 답변을 채점할 때, 평가자(에이전트)마다 기준이 달라지는
일관성 문제를 줄이고, 점수에 구체적인 근거를 붙일 수 있도록 한다.

기존 구조는 "경험 주도성 0~40점"처럼 범위만 제시되어 있어 동일한 답변을
다르게 채점할 여지가 컸다. 이를 행동 기준 평가 척도(BARS) 방식으로 교체한다.

---

## 2. 이론적 근거

### Schmidt & Hunter (1998)
Psychological Bulletin, 124(2), 262–274.

85년간의 채용 선발 연구를 메타분석한 논문.
구조화 면접의 예측 타당도 0.51, 비구조화 면접 0.38.
타당도를 높이는 조건으로 일관된 기준, 직무 관련성, 표준화된 채점을 제시.

본 설계에서의 활용:
직무 관련성이 높은 기준에 더 높은 비중을 두어야 한다는 원리의 근거로 사용.
직무별 가중치 수치는 O*NET 30.3 데이터베이스에서 조회한 직종별 KSA 중요도 점수를
정규화해서 산출한 값이다.

### Smith & Kendall (1963)
Journal of Applied Psychology, 47(2), 149–155.

행동 기준 평가 척도(BARS)의 원본 논문.
각 점수 수준에 구체적인 행동 기술(앵커)을 붙이면 평가자 간 일치도가 올라간다는
방법론적 근거를 제공.

본 설계에서의 활용:
6개 서브 기준 각각에 4단계 행동 앵커를 정의하는 직접적 근거.

### Huffcutt & Arthur (1994)
Journal of Applied Psychology, 79(2), 184–190.

면접 구조화 수준을 4단계로 분류하고 각 수준의 타당도를 실증.
Level 1(구조 없음): r ≈ 0.20 → Level 4(높은 구조): r ≈ 0.57.
구조화 수준이 높을수록 타당도가 단조 증가함을 보임.

본 설계에서의 활용:
채점 기준의 명시성과 일관성이 평가의 신뢰도를 높인다는 근거.
LLM이 "0~40점 중 몇 점인지"를 자유롭게 판단하는 방식은
구조화 수준이 낮은 방식에 해당하므로 BARS로 교체.

### McClelland (1973)
American Psychologist, 28(1), 1–14.

지식이나 지능보다 실제 직무에서 나타나는 관찰 가능한 행동(역량)이
직무 성과를 더 잘 예측한다고 주장. 역량 기반 평가와 행동 면접의 이론적 토대.

본 설계에서의 활용:
각 앵커를 "인상"이나 "느낌"이 아닌 답변에서 관찰 가능한 행동으로 정의.
"논리적으로 보인다" 같은 인상 기반 판단을 배제하고
"본인이 직접 결정했다고 기술한 문장이 있는가"처럼 행동 증거 중심으로 서술.

### O*NET 30.3 (Occupational Information Network)
미국 노동부(DOL/ETA) 운영. 직종별 요구 역량(KSA) 구성 데이터 제공.
https://www.onetonline.org/

본 설계에서의 활용:
직종별 KSA 중요도(Importance, 1~5점 척도) 점수를 조회해
에이전트별 평균 점수를 산출하고, 이를 정규화해 직무별 가중치로 사용.
데이터 조회 기준: O*NET 30.3 Database (2025년 8월 갱신).

## 3. 평가 구조 개요

### 에이전트 구성

| 에이전트 | 역할 | 평가 영역 |
|---|---|---|
| Organization | HR 담당자 | 지원동기, 직무·회사 이해도, 조직 적합성 |
| Logic | 실무 팀장 | 문제 해결 주도성, 행동 구체성, 성과 재현 가능성 |
| Technical | 현업 선임 | 기술 실사용, 기술적 선택 근거, 기술 실행 주도성 |

### 점수 구조

각 에이전트는 3개 서브 기준을 각각 0~100점으로 독립 채점한 후 평균을 낸다.

```
Organization : avg(지원동기 구체성(0~100), 직무·회사 이해도(0~100), 조직 적합성(0~100))
Logic        : avg(문제 해결 주도성(0~100), 행동 구체성(0~100), 성과 재현 가능성(0~100))
Technical    : avg(기술 실사용 여부(0~100), 기술적 선택 근거(0~100), 기술 실행 주도성(0~100))
```

### 라운드 구조

| 라운드 | 목적 | 점수 영향 |
|--------|------|----------|
| Round 0 | BARS 앵커 기반 독립 채점 | 에이전트 원점수 산출 |
| Round 1 | 다른 에이전트의 5단계 피드백 | 잠정 스탠스 부여 |
| Round 2 | 재반박 + 스탠스 갱신 | 갱신 스탠스로 원점수 조정 (-4~+4) |
| Round 3 | 제거 | — |
| Moderator | 종합 판정 | 정성 출력만 (overall, improvementTips, debateSummary) |

## 4. BARS 앵커 정의

### 채점 원칙

- 각 서브 기준에 대해 4단계 앵커 중 어느 수준에 해당하는지 먼저 판단한다.
- 판단 근거로 반드시 답변에서 직접 인용한 문장을 제시한다.
- 답변에 명시적으로 드러난 내용만 평가한다.
  추론이 필요한 경우 반드시 "추론"이라고 표시한다.
- 답변에 없는 경험, 성과, 기술 사용 여부를 임의로 보완하지 않는다.
- 인상 기반 판단 금지. "논리적으로 보인다", "성실해 보인다"는 근거가 아니다.

모든 서브 기준의 점수 범위는 0~100으로 통일한다.

```
1단계: 0~25
2단계: 26~50
3단계: 51~75
4단계: 76~100
```

---

### Organization — 지원동기 구체성 (0~100)

```
0~25점
어느 회사 지원서에도 그대로 쓸 수 있는 보편적 표현만 있다.
이 회사·직무여야 하는 이유가 전혀 없다.
예) "성장 가능성이 좋아서", "비전이 좋아서", "좋은 회사라고 생각해서"

26~50점
회사명이나 사업 분야를 언급하지만 본인과의 연결이 없다.
업계 전반에 해당하는 이유에 그친다.
예) "AI 분야라서 관심 있었다", "핀테크 시장이 성장 중이라서"

51~75점
이 회사의 특정 서비스·문화·방향을 언급하고 본인 경험과
연결하려는 시도가 있다. 다소 일반적이나 이 회사를 지향하는 맥락이 있다.

76~100점
이 회사·직무여야 하는 이유가 본인의 구체적 경험·가치관과
명확하게 연결된다. 다른 회사 지원서에 재사용할 수 없는 이 회사 특정적 답변이다.
```

---

### Organization — 직무·회사 이해도 (0~100)

```
0~25점
채용공고를 읽지 않은 것처럼 보인다.
직무 내용과 무관한 답변이거나 회사에 대한 언급이 전혀 없다.

26~50점
채용공고의 표면적 키워드만 반복한다.
예) "개발하는 회사잖아요", "백엔드 개발자 뽑는다고 해서"

51~75점
채용공고의 특정 요건이나 회사의 구체적 서비스를 언급한다.
사전 조사를 한 흔적이 있다.

76~100점
직무 요건, 회사 서비스, 업계 맥락을 연결해서 설명한다.
예) "이 직무에서 ~한 역량이 필요하다는 걸 알고 ~를 준비했다"처럼
구체적 조사 근거가 답변에 드러난다.
```

---

### Organization — 조직 적합성 (0~100)

```
0~25점
커리어 방향이 직무와 무관하거나 수혜 지향 표현만 있다.
기여할 수 있는 것에 대한 언급이 없다.
예) "많이 배우고 싶다", "성장하고 싶다"

26~50점
직무와 일부 연관된 관심·경험은 있으나 기여 방향이 불분명하다.
예) "열심히 하겠습니다"처럼 의지 선언에 그친다.

51~75점
커리어 방향이 직무와 연결되고 기여하고자 하는 구체적 방향이 있다.
다소 일반적이나 진정성이 느껴진다.

76~100점
커리어 방향, 직무 기여, 장기 성장 계획이 구체적으로 연결된다.
과거 경험에서 이를 뒷받침하는 행동이 나온다.
예) "이 팀에 ~를 기여할 수 있다"는 기여 지향 답변.
```

---

### Logic — 문제 해결 주도성 (0~100)

문제를 인식하고 판단하고 해결하는 과정에서 본인이 주도적 역할을 했는지를 본다.

```
0~25점
모든 답변의 주어가 "우리 팀이", "저희가"로만 등장한다.
본인이 직접 내린 판단이나 실행한 행동이 없다.
예) "팀이 함께 해결했어요."

26~50점
"제가 담당했다"는 나오지만 실제로 무엇을 어떻게 했는지 없다.
역할 선언에서 멈춘다.
예) "저는 프론트엔드를 맡았습니다."

51~75점
본인이 특정 행동을 했다고 나오지만 핵심 판단이나 방향 결정은
상사·팀에 귀속된다. 실행만 본인이 한 경우.
예) "팀장님 방향에 따라 제가 구현했습니다."

76~100점
상황 인식 → 판단 → 방법 선택 → 실행까지 본인 주도임이
구체적 동사로 기술된다.
예) "기존 방식의 병목을 제가 발견하고, 캐시 레이어를 직접 설계해서 적용했습니다."
```

---

### Logic — 행동 구체성 (0~100)

어떻게 판단하고 행동했는지 과정이 드러나는지를 본다.
McClelland(1973): 관찰 가능한 행동이 평가의 대상이다.

```
0~25점
결과 선언만 있다. 어떻게 했는지 과정이 전혀 없다.
예) "열심히 했다", "잘 해결했다", "성공적으로 마무리했다"

26~50점
과정이 언급되나 동사가 추상적이다. 구체적 방법이나 도구가 없다.
예) "분석했다", "논의했다", "검토했다"

51~75점
특정 방법이나 도구를 언급하며 과정을 설명한다.
다소 빠진 부분이 있으나 흐름이 파악된다.

76~100점
상황(어떤 문제였는지), 본인 행동(어떤 방법을 썼는지),
선택 이유까지 구체적으로 기술된다.
읽는 사람이 그 과정을 머릿속으로 재현할 수 있을 만큼 상세하다.
```

---

### Logic — 성과 재현 가능성 (0~100)

성과가 본인의 역량에서 비롯됐는지, 아니면 운이나 환경 덕분인지를 판단한다.

```
0~25점
성과 언급이 없거나 "좋았다", "성공했다" 수준이다.
본인 역량과 결과 사이의 인과관계가 없다.

26~50점
수치가 있으나 기간·기준·모집단이 없어 검증이 불가하다.
운이나 팀 덕분인지 본인 역량인지 구분이 안 된다.
예) "전환율이 8% 높게 나왔습니다."

51~75점
성과가 구체적이고 본인 기여가 어느 정도 연결된다.
재현 가능성 판단이 가능하나 조건이 제한적이다.

76~100점
성과 수치 + 측정 기준 + 본인 행동과의 인과관계가 명확하다.
예) "3개월간 A/B 테스트 결과 전환율이 8% 상승했고,
제가 바꾼 CTA 카피가 핵심 변수였습니다."
```

---

### Technical — 기술 실사용 여부 (0~100)

이력서나 답변에 나온 기술·툴을 실제로 사용했는지를 직무 요건과 대조해 판단한다.

```
0~25점
기술 이름 없이 "개발했다", "분석했다"만 있다.
어떤 툴·언어·프레임워크를 썼는지 전혀 언급이 없다.

26~50점
기술 이름이 나오지만 직무 요건과 연관 없는 것들만이거나,
맥락 없이 나열된다.
예) "파이썬 씁니다"

51~75점
직무 요건과 연결되는 기술을 언급하고 사용 맥락이 있다.
어느 프로젝트에서 썼는지 정도는 파악된다.

76~100점
직무 요건의 기술을 실제 프로젝트 맥락에서 구체적으로 설명한다.
예) "결제 모듈에서 동시성 문제를 해결하기 위해 Redis를 직접 도입했습니다."
```

---

### Technical — 기술적 선택 근거 (0~100)

왜 그 기술을 썼는지 이유를 설명할 수 있는지를 본다.

```
0~25점
기술을 왜 썼는지 이유가 전혀 없다.
예) "그냥 썼어요", "팀에서 원래 쓰던 거라서"

26~50점
이유가 있으나 매우 일반적이다. 대안을 고려하지 않았다.
예) "빠르니까", "쉬우니까"

51~75점
특정 문제·상황을 언급하며 선택 이유를 설명한다.
대안과의 비교는 없지만 의식적 선택의 흔적이 있다.

76~100점
문제 상황 + 대안 비교 + 선택 이유가 명확하다.
예) "MySQL 대신 PostgreSQL을 선택한 이유는
JSON 컬럼 쿼리 성능 차이 때문이었습니다."
```

---

### Technical — 기술 실행 주도성 (0~100)

기술 선택, 구현, 트러블슈팅에서 본인이 얼마나 직접 판단하고 실행했는지를 본다.
Logic의 문제 해결 주도성과 이름이 유사하지만 평가 대상이 다르다.
이 항목은 기술 실행 영역에만 집중한다.

```
0~25점
팀이 결정한 것을 실행만 했음이 명확하다.
본인 역할이 없거나 완전히 수동적이다.

26~50점
역할이 있으나 지시에 따른 수행이 대부분이다.

51~75점
특정 기술 영역에서 본인이 판단하고 실행한 것이 있다.

76~100점
기술적 방향, 구현 방법, 트러블슈팅을 본인이 주도했다.
예) "제가 아키텍처를 제안하고, 팀 설득 후 직접 구현했습니다."
```

## 5. 점수 집계 알고리즘

### Step 1 — 에이전트별 원점수 (Round 0)

BARS 앵커 기준으로 서브 기준별 0~100 독립 채점 후 평균.

```
org_r0   = avg(지원동기(0~100), 이해도(0~100), 적합성(0~100))
logic_r0 = avg(주도성(0~100), 구체성(0~100), 재현성(0~100))
tech_r0  = avg(실사용(0~100), 근거(0~100), 주도성(0~100))
```

### Step 2 — Round 1 잠정 스탠스 부여

각 에이전트의 원점수에 대해 다른 두 에이전트가 5단계 잠정 스탠스를 부여한다.

```
매우 동의  : +2
동의       : +1
중립       :  0
비동의     : -1
매우 비동의: -2
```

Round 1 스탠스는 잠정값이며 점수에 즉시 반영되지 않는다.

### Step 3 — Round 2 재반박 + 스탠스 갱신

재반박을 들은 비판자가 Round 1 스탠스를 갱신한다.
갱신 스탠스가 Round 1 잠정 스탠스를 대체하며, 최종 조정에는 Round 2 갱신 스탠스만 사용한다.

반박이 설득력 있으면 비판자가 스탠스를 완화하고, 설득에 실패하면 유지 또는 강화한다.

```
org_adjusted   = clamp(org_r0   + (logic→org 갱신스탠스) + (tech→org 갱신스탠스),   0, 100)
logic_adjusted = clamp(logic_r0 + (org→logic 갱신스탠스) + (tech→logic 갱신스탠스), 0, 100)
tech_adjusted  = clamp(tech_r0  + (org→tech 갱신스탠스)  + (logic→tech 갱신스탠스), 0, 100)
```

조정 후 점수가 0 미만이면 0, 100 초과면 100으로 강제 적용한다.

### Step 4 — 직무 기반 가중치 적용

O*NET 30.3 데이터베이스의 직종별 KSA 중요도 점수를 조회해 정규화한 값.

에이전트-KSA 매핑:
- Organization: Customer and Personal Service, Personnel and Human Resources,
  Administration and Management, Social Perceptiveness, Coordination,
  Service Orientation, Oral/Written Expression, Oral/Written Comprehension
- Logic: Critical Thinking, Complex Problem Solving, Judgment and Decision Making,
  Active Learning, Deductive Reasoning, Inductive Reasoning,
  Problem Sensitivity, Information Ordering
- Technical: Computers and Electronics, Engineering and Technology, Mathematics,
  Programming, Technology Design, Mathematical Reasoning, Number Facility

```
weightedScore = org_adjusted * w_org + logic_adjusted * w_logic + tech_adjusted * w_tech
```

### Step 5 — 최종 추천 레벨

Moderator LLM이 판단하지 않는다. weightedScore로 코드에서 직접 산출한다.

```
weightedScore = clamp(weightedScore, 0, 100)

90 이상       → 강력 추천
75 이상 90 미만 → 추천
45 이상 75 미만 → 보류
44 이하        → 비추천
```

## 6. 평가 프롬프트 구조

Huffcutt & Arthur(1994): 채점 기준이 명시적일수록 타당도가 높아진다.

기존 방식: LLM이 "0~40점 중 몇 점인지" 자유롭게 판단
변경 방식: LLM이 4개 앵커 중 해당 수준을 먼저 판단 후 구간 내 점수 산출

```
[채점 지시 예시]

아래 4단계 앵커를 읽고 지원자 답변이 어느 단계에 해당하는지 판단하세요.

문제 해결 주도성:
- 0~25점: [앵커 텍스트]
- 26~50점: [앵커 텍스트]
- 51~75점: [앵커 텍스트]
- 76~100점: [앵커 텍스트]

판단 시 반드시 지켜야 할 조건:
1. 답변에서 직접 인용한 문장을 근거로 제시할 것
2. 답변에 드러나지 않은 내용을 임의로 추론하지 말 것
3. 추론이 필요한 경우 반드시 "추론"이라고 표시할 것
4. "논리적으로 보인다" 같은 인상 기반 표현은 근거로 쓰지 말 것
```

Round 1 피드백 프롬프트 (잠정 스탠스):

```
[피드백 지시 예시]

상대 에이전트의 평가와 Round 0 점수를 검토하고 아래 5단계 중 하나로 스탠스를 표시하세요.

매우 동의  (+2): 평가가 매우 적절하다. 놓친 부분이 없다.
동의       (+1): 평가가 대체로 적절하다.
중립        (0): 동의도 비동의도 아니다.
비동의     (-1): 평가에 동의하지 않는다. 구체적 근거 필수.
매우 비동의(-2): 평가가 크게 잘못됐다. 구체적 근거 필수.

비동의(-1, -2)의 경우 반드시 답변 텍스트에서 직접 인용한 근거를 제시할 것.
인용 없는 비동의는 무효 처리.
```

Round 2 재반박 + 스탠스 갱신 프롬프트:

```
[재반박 및 스탠스 갱신 지시 예시]

상대 에이전트의 반박을 읽고 Round 1에서 부여한 스탠스를 갱신하세요.
갱신 스탠스가 최종 점수 조정에 반영됩니다.

반박이 설득력 있다면 스탠스를 완화하고, 그렇지 않다면 유지 또는 강화하세요.
스탠스 변경 여부와 무관하게 반드시 이유를 제시해야 합니다.

매우 동의  (+2): 반박을 듣고 보니 평가가 매우 적절했다.
동의       (+1): 반박이 타당하다. 기존 스탠스를 완화한다.
중립        (0): 반박이 일부 타당하나 전체적으로 판단이 바뀌지 않는다.
비동의     (-1): 반박이 설득력 없다. 기존 비동의를 유지한다.
매우 비동의(-2): 반박이 오히려 기존 판단을 강화한다.
```

## 7. 미결 사항

모든 미결 사항이 해소되었다.

1. ~~BARS 점수 기준 라운드~~ → 해소
   Round 0 BARS 원점수 + Round 2 갱신 스탠스 합산으로 조정 점수 산출.
   Round 3(AgentFinalOpinion) 제거.

2. ~~직무별 가중치 검증~~ → 해소
   O*NET 30.3 데이터베이스 기반으로 산출. 운영 중 보정이 필요할 경우
   실제 평가 데이터를 축적한 후 재산출한다.

3. ~~난이도 보정 필요 여부~~ → 해소
   난이도 보정을 적용하지 않는다.
   BARS는 답변 내용 기반으로 작동하므로 난이도와 무관하게 일관된 기준이 유지된다.

4. ~~점수 범위 클램핑~~ → 해소
   조정 후 에이전트 점수 및 weightedScore를 0~100으로 강제 적용한다.

5. ~~recommendLevel 산출 주체~~ → 해소
   weightedScore 기준으로 코드에서 직접 산출. Moderator LLM 판단 없음.

6. ~~Moderator 출력 구성~~ → 해소
   overall(strengths, weaknesses, advice), improvementTips(3개), debateSummary 유지.
   점수 산출 역할은 제거.
