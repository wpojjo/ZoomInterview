# 서비스 아키텍처 및 데이터 흐름

## 전체 흐름

```
채용공고 입력
    │
    ▼
채용공고 분석 (Jina + OCR + LLM)
    │
    ├──────────────────────────────────┐
    │                                  │
    ▼ (비동기)                         ▼ (면접 시작 시)
DART 기업정보 수집                    뉴스 크롤링
(6개월 캐싱)                          (normal/hard만)
    │                                  │
    └──────────────┬───────────────────┘
                   │
                   ▼
            면접 진행 (3 에이전트)
            Organization → Logic → Technical
            각 에이전트: 기본 질문 → 꼬리질문(0~3회)
                   │
                   ▼
            토론 (4 라운드)
            Round 0: 독립 평가
            Round 1: 상호 피드백
            Round 2: 재반박
            Round 3: 최종 의견
                   │
                   ▼
            중재자 종합 점수 및 피드백
```

## 1. 채용공고 분석

**API**: `POST /api/job-posting/analyze`

사용자가 채용공고 URL 또는 텍스트를 입력하면 3단계로 분석합니다.

**Step 1 — 텍스트 추출**
- URL 입력 시: Jina API(`r.jina.ai/{url}`)로 마크다운 변환 (8,000자 제한)
- 실패 시: HTML 직접 파싱 후 이미지 URL 추출

**Step 2 — 이미지 OCR**
- 이미지 URL 최대 3개를 Google Cloud Vision으로 텍스트 추출 (스크린샷/PDF 대응)

**Step 3 — LLM 파싱**
- RunPod LLM 호출 → 아래 필드 추출
- 텍스트와 OCR 결과를 각각 분석 후 더 긴 문자열로 병합

추출 필드:

| 필드 | 설명 | 사용 에이전트 |
|------|------|-------------|
| `responsibilities` | 담당업무 | 전체 |
| `requirements` | 자격요건 | 전체 |
| `preferredQuals` | 우대사항 | 전체 |
| `companyName` | 회사명 | 전체 |
| `divisionName` | 사업부/팀명 | 전체 |
| `techStack` | 기술스택 | Technical |
| `companyDescription` | 회사 소개 | Organization |
| `companyCulture` | 조직 문화 | Organization |

분석 완료 후 `collectCompanyInfo(jobPostingId, companyName)`를 비동기로 호출해 DART 수집을 시작합니다.

## 2. DART 기업정보 수집

**함수**: `lib/company-info-collector.ts` → `collectCompanyInfo()`

채용공고 분석 완료 직후 백그라운드에서 실행됩니다. 사용자 대기 없이 진행됩니다.

**회사명 정규화**
- 브랜드명 → 법인명 매핑 48개 (`BRAND_MAP`)
  - 예: "토스" → "비바리퍼블리카", "배달의민족" → "우아한형제들"

**캐싱**
- `company_cache` 테이블에 6개월 캐시 유지
- 캐시 존재 시 DART API 호출 생략

**수집 항목** (병렬 호출):

| 필드 | 출처 | 내용 | 사용 에이전트 |
|------|------|------|-------------|
| `foundedYear` | `/company.json` | 설립연도 | Organization |
| `listingStatus` | `/company.json` | 코스피/코스닥/코넥스/비상장 | Organization |
| `financialSummary` | `/fnlttSinglAcnt.json` | 3년 매출·영업이익 및 전년비 | Organization |
| `employeeSummary` | `/empSttus.json` | 임직원 수·평균근속·평균연봉 | Organization |
| `recentDisclosures` | `/list.json` | 최근 180일 인수·합병·투자·신사업 공시 (최대 5건) | Organization |
| `businessSummary` | 사업보고서 HTML | 사업 개요·주요 제품·서비스 LLM 요약 (수집 시 변환) | Organization |

수집 후 `company_cache` upsert → `company_info`로 `job_postings`와 연결합니다.

## 3. 뉴스 크롤링

**함수**: `lib/naver-news-crawler.ts`

면접 질문 생성 시 트리거됩니다. 난이도가 `normal` / `hard`일 때만 동작합니다.

**Step 1 — 직무 분류 감지**
- `responsibilities + requirements` 텍스트에서 21개 직무 분류 중 하나를 키워드 매칭으로 감지
- 예: "백엔드 개발, 데이터 파이프라인" → `IT개발·데이터`

**Step 2 — 검색 쿼리 생성**
- 분류별 사전 정의 키워드 (최대 4개): 예) "삼성전자 신사업 진출", "삼성전자 M&A"
- 담당업무·기술스택에서 추출한 키워드 (최대 4개): 예) "삼성전자 MSA", "삼성전자 클라우드"

**Step 3 — 네이버 뉴스 API 호출**
- 검색어당 2건, 유사도 정렬
- HTML 태그·엔티티 제거 후 정제

**Step 4 — LLM 관련성 필터링**
- 직무별 주요 쟁점 목록과 대조해 관련 기사만 선별
- 주가·인사발령·홍보성 기사 제거
- 최대 6건 반환, 기사마다 `matchedIssue` 태깅

프롬프트 주입 형태:
```
[삼성전자 - IT개발·데이터 관련 최근 뉴스]

주요 이슈:
• 신기술 도입 (AI, 클라우드 전환)
• 보안 사고·개인정보 유출

최근 뉴스:
• 삼성, AI 칩 설계 속도 높인다
  └ 관련 쟁점: 신기술 도입 (AI)
  출처: news.samsung.com (2024-03-15)
```

## 4. 면접 진행

### 에이전트 구조

3명의 AI 면접관이 순차적으로 질문합니다.

| 에이전트 | 역할 | 판단 기준 |
|--------|------|---------|
| **Organization** (HR 담당자) | 지원동기, 조직 적합성, 장기 재직 가능성 | "왜 하필 우리 회사인가?" |
| **Logic** (실무 팀장) | 경험 주도성, 행동 구체성, 성과 재현 가능성 | "내 팀에서 실제로 어떻게 일할 사람인가?" |
| **Technical** (현업 선임) | 기술 실사용, 선택 근거, 직무 요건 연결 | "이력서에 써놓은 것들, 실제로 아는 건가?" |

### 질문 생성

**API**: `POST /api/interview/question`

- Organization 첫 질문은 고정: "간단한 자기소개와 지원동기를 말씀해주세요."
- 이후 질문은 LLM이 면접 대화 히스토리 + 지원자 프로필 + 채용공고 + DART/뉴스 컨텍스트를 기반으로 동적 생성
- `normal` / `hard` 난이도에서 뉴스 컨텍스트 주입 (뉴스를 직접 언급하지 않고 지원자 경험과 연결하도록 지시)

### 속마음 및 꼬리질문

**API**: `POST /api/interview/follow-up`

각 답변 후 에이전트의 속마음 3개 항목을 생성합니다:
- `reaction`: 듣는 순간 첫 반응
- `judgment`: 머릿속 판단
- `curiosity`: 더 보고 싶은 것

꼬리질문 최대 횟수 (난이도별):
- `tutorial`: 0회 / `easy`: 1회 / `normal`: 2회 / `hard`: 3회

꼬리질문 트리거 기준 (예시):
- Organization: 경험이 추상적, 수혜 지향("배우고 싶다") 표현
- Logic: 수치 근거 부재, "우리 팀이"와 "내가" 불구분
- Technical: 기술명만 언급, 선택 이유 미설명

## 5. 토론 및 최종 평가

**API**: `POST /api/interview/debate`

토론은 백그라운드에서 진행되며, 클라이언트는 `GET /api/interview/debate/{sessionId}/status` 폴링으로 실시간 확인합니다.

### 라운드 구조

| 라운드 | 목적 | 점수 영향 |
|--------|------|----------|
| Round 0 | BARS 앵커 기반 독립 채점 | 에이전트 원점수 산출 |
| Round 1 | 5단계 스탠스 피드백 (-2~+2) | 잠정 스탠스 부여 (즉시 반영 안 됨) |
| Round 2a | 재반박 — 평가받은 에이전트가 비판에 응답 | — |
| Round 2b | 스탠스 갱신 — 비판자가 재반박을 읽고 스탠스 업데이트 | 갱신 스탠스로 원점수 조정 (-4~+4) |
| Moderator | 정성 종합 (overall, tips, debateSummary) | 점수 산출 없음 |

**Round 0 — BARS 독립 채점**

각 에이전트가 3개 서브 기준을 0~100으로 독립 채점 후 평균을 에이전트 점수로 사용합니다. Smith & Kendall(1963) BARS 방식으로 4단계 행동 앵커를 명시해 LLM의 자유 채점을 구조화합니다.

| 에이전트 | 서브 기준 (각 0~100) |
|--------|-------------------|
| Organization | 지원동기 구체성 / 직무·회사 이해도 / 조직 적합성 |
| Logic | 문제 해결 주도성 / 행동 구체성 / 성과 재현 가능성 |
| Technical | 기술 실사용 여부 / 기술적 선택 근거 / 기술 실행 주도성 |

앵커 예시 (지원동기 구체성):
- 0~25: 어느 회사에도 쓸 수 있는 보편적 표현만. 예) "성장 가능성이 좋아서"
- 26~50: 회사명·사업 분야 언급하나 본인과의 연결 없음
- 51~75: 이 회사의 특정 서비스·문화를 언급하고 본인 경험과 연결 시도
- 76~100: 이 회사여야 하는 이유가 본인 경험·가치관과 명확히 연결

**Round 1 — 5단계 스탠스 피드백**

각 에이전트가 다른 2명의 평가에 숫자 스탠스와 코멘트를 부여합니다.

| 스탠스 | 의미 | 근거 조건 |
|--------|------|----------|
| +2 | 매우 동의 | — |
| +1 | 동의 | — |
| 0 | 중립 | — |
| -1 | 비동의 | 답변 직접 인용 필수 |
| -2 | 강한 비동의 | 답변 직접 인용 필수 |

Round 1 스탠스는 잠정값이며 점수에 즉시 반영되지 않습니다.

**Round 2a — 재반박**

평가받은 에이전트(defendant)가 자신을 향한 비판에 반박 또는 수용합니다.

**Round 2b — 스탠스 갱신**

비판한 에이전트(critic)가 재반박을 읽고 Round 1 스탠스를 갱신합니다. 갱신 스탠스가 Round 1 잠정 스탠스를 대체하며, 최종 점수 조정에는 갱신 스탠스만 사용합니다.

### 최종 점수 산출

**Step 1 — 에이전트별 조정 점수**

```
org_adjusted   = clamp(org_r0   + (logic→org 갱신스탠스) + (tech→org 갱신스탠스),   0, 100)
logic_adjusted = clamp(logic_r0 + (org→logic 갱신스탠스) + (tech→logic 갱신스탠스), 0, 100)
tech_adjusted  = clamp(tech_r0  + (org→tech 갱신스탠스)  + (logic→tech 갱신스탠스), 0, 100)
```

**Step 2 — 직무별 가중 점수**

O*NET 30.3 직종별 KSA 중요도 점수를 정규화한 값:

| 직무 분류 | Organization | Logic | Technical |
|---------|-------------|-------|-----------|
| 개발 / 데이터 / AI | 32% | 38% | 30% |
| 기획 / PM | 37% | 36% | 27% |
| 디자인 / UX | 29% | 38% | 33% |
| 영업 / 마케팅 | 39% | 38% | 23% |
| HR / 경영지원 | 43% | 38% | 19% |
| 연구 / 분석 | 28% | 38% | 34% |
| 분류 불명 | 33% | 34% | 33% |

```
weightedScore = org_adjusted × w_org + logic_adjusted × w_logic + tech_adjusted × w_tech
```

**Step 3 — 불일치 시그널**

```
stddev = std(org_adjusted, logic_adjusted, tech_adjusted)

stddev < 10     → 평가 일치
10 이상 20 미만 → 부분 불일치
20 이상         → 평가자 간 의견 불일치
```

불일치가 크다고 점수를 조정하지 않습니다. 투명하게 사용자에게 전달하는 것이 목적입니다.

**Step 4 — 추천 레벨 (코드 직접 산출)**

```
90 이상         → 강력 추천
75 이상 90 미만 → 추천
45 이상 75 미만 → 보류
44 이하         → 비추천
```

추천 레벨은 LLM이 판단하지 않습니다. `weightedScore` 기준으로 코드에서 직접 분기합니다.

## 6. 주요 설계 결정

**DART 데이터 비동기 수집**
채용공고 분석 완료 직후 fire-and-forget으로 실행. 면접 시작 전까지 데이터가 준비됩니다.

**토론 백그라운드 처리**
4라운드 토론은 LLM 호출이 많아 시간이 걸립니다. 백그라운드 실행 + 폴링으로 사용자에게 진행 상황을 실시간 표시합니다.

**뉴스 활용 방식**
뉴스를 "알고 있냐"고 직접 묻지 않고, 지원자의 경험·판단과 연결하도록 LLM에 지시합니다.

**BARS vs 토론 구조의 충돌 해소**
BARS의 전제는 평가자 독립성(Smith & Kendall, 1963)이다. 토론이 점수에 영향을 주면 순수한 BARS 점수가 아니게 된다. 이 충돌을 다음 방식으로 해소했다: Round 0에서 BARS 기준으로 독립 채점한 원점수를 산출하고, Round 2b 갱신 스탠스(-2~+2)만 점수 조정에 반영한다(-4~+4 범위). 원점수는 BARS 원칙을 따르고, 토론은 그 원점수를 소폭 조정하는 역할에 그친다.

**Round 3 제거**
기존 Round 3(최종 의견 재채점)는 에이전트가 토론 후 점수를 자유롭게 바꿀 수 있어 BARS 일관성을 깼다. 갱신 스탠스(-4~+4) 조정으로 대체해 토론의 영향을 정량화하고 범위를 제한했다.

**추천 레벨 코드 직접 산출**
기존 Moderator LLM이 recommendLevel을 결정했으나 동일한 점수에 다른 레벨을 출력하는 비일관성 문제가 있었다. weightedScore 기준 코드 분기로 교체해 일관성을 확보했다.

**직무별 가중치 근거**
단순 균등 분할(33/33/34%)은 직무마다 요구 역량 비중이 다르다는 현실을 반영하지 못한다. O*NET 30.3 데이터베이스의 직종별 KSA 중요도(Importance) 점수를 조회해 에이전트-KSA 매핑을 통해 가중치를 정규화했다. 한국 직무 카테고리와 O*NET 직종 코드 간 1:1 매핑이 어려운 경우 근사 직종 코드를 사용했다.

**에이전트별 데이터 비대칭**
각 에이전트의 역할에 맞는 컨텍스트만 주입합니다. Organization은 "왜 이 회사인가"를 검증해야 하므로 회사 전반 정보(재무·문화·사업보고서 포함)와 다트 힌트 4종을 받습니다. Logic은 실무 경험 검증이 목적이므로 회사명·사업부명·직무 분류만 받습니다(재무·공시 등은 평가 기준과 무관). Technical은 기술 검증이 목적이므로 회사명·사업부명·직무 분류·기술스택을 받습니다. 중재자는 3개 에이전트의 final opinion을 종합하는 역할이므로 회사명만 받습니다(회사 맥락은 이미 에이전트 opinion에 녹아 있어 중복 주입을 피합니다). 직무 분류(`jobClassification`)는 채용공고 텍스트에서 감지한 21개 분류로, Logic·Technical 에이전트가 해당 직무의 현업 입장에서 평가하도록 유도합니다. 질문 생성·속마음·토론 전 단계에 동일한 분리 원칙을 적용합니다.

**사업보고서 LLM 요약**
DART에서 사업보고서 HTML 수집 시 즉시 LLM 요약(3문장 이내)으로 변환 후 DB에 저장합니다. 원문은 최대 6,000자였으나 요약 후 ~600자로 축소되어 Organization 에이전트 컨텍스트 크기를 줄입니다. 요약 실패 시 원문 500자 슬라이스로 폴백합니다.

**Organization 페르소나 다트 힌트 4종**
Organization 에이전트에만 다트 데이터 기반 질문 방향 힌트를 조건부 주입합니다. 추가 LLM 호출 없이 정규식·필드 조합으로 처리하며, 힌트 톤은 "왜 이 회사인가" 평가 철학에 맞춰 역량 평가가 아닌 조직 적합성·지원동기 검증 방향으로 작성됩니다.

**1) 재무 국면 — `financialSummary` 파싱**

| 국면 | 조건 | 질문 방향 |
|------|------|-----------|
| 확장기 | 매출 +10% 이상, 또는 매출 +20% 이상 + 영업 적자(스케일업) | 변화·확장 환경 선호 검증, 안정 지향 답변 파고들기 |
| 위기 | 매출 감소 + 흑자 | 회사 도전 과제 인지 여부, '왜 이 시점에' 정합성 |
| 생존모드 | 매출 감소 + 적자, 또는 매출 정체 + 적자 | 상황 인지 후 합류 이유, 단기 압박 환경 준비도 |

매출 고성장 + 영업 적자(쿠팡·토스 등 스케일업)는 확장기로 분류합니다. 단순 적자 분류는 의도적 손실과 진짜 위기를 구분하지 못해 미스매치가 큽니다.

**2) 기업 성숙도 — `foundedYear` + `listingStatus` + `employeeSummary` 조합**

| 단계 | 조건 | 질문 방향 |
|------|------|-----------|
| 초기 | 설립 8년 이하 + 비상장 + 임직원 300명 이하 | 자기주도성, 역할 경계 유연성, 리소스 부족 속 성과 |
| 성숙기 | 설립 15년 이상 + 상장 + 임직원 2000명 이상 | 대규모 조직 협업, 절차·문서 기반 실행, 전문성 깊이 |

세 필드 중 하나라도 결측이면 분류 포기. 양 극단만 명확할 때만 힌트를 주입해 노이즈를 줄입니다. 설립연도 단독으론 합병·분사 케이스에서 misleading하므로 상장 여부·임직원 수로 보강합니다.

**3) 최근 공시 시그널 — `recentDisclosures` 키워드 감지**

- 인수·합병 감지: 조직 변화·통합 과정의 실행력, 불확실성 속 우선순위 판단 위주 파악
- 신사업·투자 감지: 검증되지 않은 영역의 학습·실행 능력, 정답 없는 문제 처리 위주 파악

`recentDisclosures` 자체가 4개 키워드("인수","합병","투자","신사업")로 이미 필터링된 결과이므로 단순 존재 여부만 검사합니다.

**4) 사업 요약 활용 지침 — `businessSummary` 존재 시 항상 주입**

지원자가 "왜 이 회사인가" 답변에서 사업 방향·고객군·서비스를 실제로 참조하는지 검증하도록 지시합니다. 보편적 답변("성장 가능성", "비전 공감")에 그치고 사업 특성을 한 번도 언급하지 않으면 사업 요약의 특정 문구를 인용하며 파고들도록 합니다. 분류 없이 평가 앵커로 사용합니다.
